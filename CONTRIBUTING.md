## Getting Started

### Prerequisites

- **Node.js** 20+ (LTS recommended)
- **pnpm** 9+

### Setup

1. Fork the repository and clone your fork:
   ```bash
   git clone https://github.com/<your-username>/KeyZen.git
   cd KeyZen
   ```
2. Install dependencies:
   ```bash
   pnpm install
   ```
3. Start the development server:
   ```bash
   pnpm dev
   ```
4. Open [http://localhost:3000](http://localhost:3000).

## Code Quality

- **TypeScript is required.** Do not use `any` — use proper types or generics.
- **No comments explaining _what_ the code does.** Good names do that. Only add a comment when explaining _why_ something exists (a workaround, a hidden invariant, a browser quirk).
- **No multi-line comment blocks or long docstrings.**
- **No speculative abstractions.** Three similar lines of code is better than a premature helper. Only extract a function when it is used in at least two distinct places.
- **No error handling for impossible cases.** Trust TypeScript, Next.js, and framework guarantees. Only validate at system boundaries (user input, fetched JSON).
- **No new `console.log` statements in production code.**
- **No `useEffect` unless the no-use-effect rules allow it.** Prefer derived state, event handlers, data-fetching libraries, or `useMountEffect` for the rare mount-only external sync case. See `.claude/skills/no-use-effect.md` for the full rule set.
- Run the full check suite before opening a PR:

```bash
pnpm typecheck   # TypeScript — must pass with zero errors
pnpm lint        # ESLint — must pass with zero warnings
pnpm format      # Prettier — auto-formats .ts/.tsx files
```

## Data Files — What Not to Edit Manually

Several files in this repo are **generated automatically**. Editing them by hand will be overwritten on the next `pnpm install` or will introduce format inconsistencies.

### `data/quotes.json`

**Do not edit this file.**

It is generated by the `postinstall` script from the `inspirational-quotes` npm package. If you need to change the quotes, either update the package version or replace the package in `package.json` and update the `postinstall` script accordingly.

### `lib/code/data.json`

**Do not edit this file.**

It is generated by `scripts/generate-code-data.mjs` at install time. This file is a compiled manifest of all code snippets. To add or change code snippets, edit the source files inside the `data/` directory (e.g. `data/javascript/`, `data/python/`), then re-run:

```bash
node scripts/generate-code-data.mjs
```

### `public/sw.js`

**Do not edit this file.**

It is the compiled output of `app/sw.ts` and is produced by the Next.js build. Edit `app/sw.ts` for service worker logic changes, then run `pnpm build` to regenerate it.

## Scripts and When to Run Them

| Script | When to run |
|--------|-------------|
| `pnpm install` | After cloning, or after changing `package.json` |
| `pnpm dev` | Local development (port 3000) |
| `pnpm build` | Before testing production behaviour or deploying |
| `pnpm start` | Serve the production build locally |
| `pnpm typecheck` | Before every PR — must pass zero errors |
| `pnpm lint` | Before every PR — must pass zero warnings |
| `pnpm format` | Before every PR — auto-fixes formatting |
| `node scripts/generate-code-data.mjs` | After adding or editing files inside `data/` |

The `postinstall` hook runs automatically after `pnpm install`. It does two things:

1. Reads `node_modules/inspirational-quotes/data/data.json`, strips the BOM character, and writes the result to `data/quotes.json`.
2. Runs `scripts/generate-code-data.mjs` to compile code snippet manifests into `lib/code/data.json`.

You should not need to run these manually unless you are making changes to the data pipeline.

## Service Worker and Offline Support

KeyZen ships as a PWA. The service worker is built on [Serwist](https://serwist.pages.dev), a Next.js-compatible service worker library.

### How it works

- **Source:** `app/sw.ts`
- **Output:** `public/sw.js` (built by `pnpm build`, do not edit directly)
- **Config:** `next.config.mjs` — sets the SW entry point, output filename, and options

Key behaviour:

- `skipWaiting: true` — a new service worker activates immediately without waiting for the old one to be unloaded.
- `clientsClaim: true` — the new SW takes control of all open tabs instantly.
- `precacheEntries` — all static assets (JS, CSS, fonts, images, sound files) are pre-cached at install time using the Serwist `__SW_MANIFEST` injection.
- `reloadOnOnline: true` — the page reloads automatically when the device comes back online.
- The service worker is **disabled in development** (`NODE_ENV !== "production"`). You will not see offline behaviour locally unless you run `pnpm build && pnpm start`.

### What to be careful about

- Do not add large binary assets to the project without considering their impact on the precache size. Every file in `public/` that is referenced in the build will be precached.
- Sound pack files are large. New sound packs add to the precache size. If you add sounds, verify the total precache payload stays reasonable.
- Changes to `app/sw.ts` require a production build to take effect.

## Sound System — Adding Audio and New Keyboard Sounds

### How the sound system works

KeyZen supports multiple keyboard sound packs. There are two types:

**Single-file (default):** One `.ogg` file plays on every keypress.

**Sprite-mapped packs:** A single audio file combined with a `config.json` that maps each key code to an `[offset_ms, duration_ms]` pair. This lets different keys produce different sounds from one audio file.

### File locations

```
public/sounds/
├── sound.ogg              ← default sound pack (single file)
├── click-sound.wav        ← UI button click feedback (separate from typing sounds)
├── cherrymx-black-pbt/
│   ├── sound.ogg          ← audio sprite
│   └── config.json        ← key-to-offset mapping
├── cherrymx-blue-pbt/
│   └── ...
└── ...
```

### Adding a new single-file sound

1. Add your `.ogg` or `.wav` file to `public/sounds/`.
2. Open `lib/settings-data.ts`.
3. Add an entry to the `SOUND_PACKS` array:

```typescript
{
  id: "my-pack",       // must match the SoundPack union type
  label: "My Pack",
  url: "/sounds/my-sound.ogg",
}
```

4. Add `"my-pack"` to the `SoundPack` type union at the top of the file.

### Adding a new sprite-mapped sound pack

1. Create a subdirectory under `public/sounds/` (e.g. `public/sounds/my-pack/`).
2. Place the audio sprite as `sound.ogg` inside that directory.
3. Create `config.json` in the same directory:

```json
{
  "name": "My Pack",
  "id": "my-pack",
  "keyDefType": "code",
  "defines": {
    "KeyA": [0, 80],
    "KeyB": [100, 80],
    "Space": [200, 100]
  }
}
```

- `keyDefType`: use `"code"` (maps to `KeyboardEvent.code`)
- `defines`: map each key code to `[start_offset_ms, duration_ms]`

4. Register in `lib/settings-data.ts`:

```typescript
{
  id: "my-pack",
  label: "My Pack",
  url: "/sounds/my-pack/sound.ogg",
  configUrl: "/sounds/my-pack/config.json",
}
```

5. Add `"my-pack"` to the `SoundPack` union type.

### Click sound

The UI click sound (`public/sounds/click-sound.wav`) plays on all `button`, `a`, `[role="button"]`, and `[role="switch"]` elements. To suppress it on a specific element, add `data-no-click-sound` to the element. Do not change the click sound file without replacing it with something at a similar volume and duration.

## Adding New Languages

Word lists live in `public/languages/`. KeyZen uses MonkeyType-compatible JSON format.

### File format

```json
{
  "words": ["the", "be", "of", "and", "a", "to", "in"]
}
```

### Steps to add a language

1. Create `public/languages/{code}.json` with the format above. Use lowercase, no spaces (e.g. `spanish.json`, `hindi.json`).
2. Optionally create `public/languages/{code}_1k.json` — this is the "hard" variant. Words here should be longer or less common (the loader targets 5–12 character words for hard mode). If you do not create a `_1k` file, hard mode falls back to the base list.
3. Add an entry to `public/languages/_manifest.json`:

```json
{ "name": "Spanish", "code": "spanish", "has1k": true }
```

Set `"has1k": false` if you did not create a `_1k` variant.

### RTL languages

If your language reads right-to-left, add its code to the `RTL_CODES` array in `lib/languages.ts`. The typing interface will automatically mirror correctly.

Arabic word lists benefit from diacritic stripping. The `stripArabicDiacritics` function in `lib/languages.ts` already handles harakat — make sure your word list does not depend on diacritics to be readable.

### Fallback behaviour

If a language file cannot be fetched, the code falls back to the `random-words` npm package (English words). This is only a safety net — always include a proper word list.

### Minimum requirements

- At least 200 words in the base list.
- Words should be lowercase (unless the language requires capitalisation).
- No punctuation inside words — punctuation is applied by the test engine as a modifier.

## Word and Quote Packages

### Words

- **Primary:** Language-specific JSON files in `public/languages/` (fetched on demand, cached in memory for the session).
- **Fallback:** [`random-words`](https://www.npmjs.com/package/random-words) npm package — used only when a language file is unavailable.
- **Modifiers** (`lib/words.ts`): Punctuation and number insertion are applied as post-processing over any word list. Do not bake punctuation into word list files.

### Quotes

- **Source:** [`inspirational-quotes`](https://www.npmjs.com/package/inspirational-quotes) npm package.
- **Built file:** `data/quotes.json` — generated at install time. Do not edit this file by hand.
- **Selection** (`lib/quotes.ts`): Quotes are filtered by character length — short (40–130), medium (131–199), long (200–600) — then randomly sampled.
- To add quotes: update the npm package or replace it and adjust the `postinstall` script in `package.json`.

### Code snippets

- **Sources:** Files inside `data/{language}/` directories (e.g. `data/javascript/`, `data/python/`).
- **Built file:** `lib/code/data.json` — generated by `scripts/generate-code-data.mjs`. Do not edit this file by hand.
- To add a code snippet: add a source file to the appropriate language directory, then run `node scripts/generate-code-data.mjs`.
- Syntax highlighting uses [Shiki](https://shiki.style). Supported languages follow Shiki's grammar list.

## How Result Validation Works

Result validation lives in `lib/validate-result.ts` and `lib/wpm-count.ts`. It runs client-side at the end of every test and rejects results that are invalid or indicate automation.

### WPM calculation

```
WPM = (correct_word_chars + correct_spaces) / 5 / elapsed_minutes
```

The divisor `5` is the standard assumed word length. Only fully correct words count toward WPM; partially correct words do not. Raw WPM counts all typed characters regardless of correctness.

### Accuracy

```
accuracy = allCorrectChars / (allCorrectChars + incorrectChars + missedChars) × 100
```

### Consistency

Consistency is calculated as the inverse coefficient of variation of per-second WPM samples:

```
consistency = 100 − (σ / μ × 100)
```

100 means perfectly consistent typing. Very high consistency at high WPM is a bot-detection signal.

### Rejection conditions

A result is rejected if any of the following are true:

| Condition | Reason |
|-----------|--------|
| `no_keystrokes` | Zero keys were pressed |
| `invalid_numbers` | WPM, raw, or accuracy is `NaN` or `Infinity` |
| `invalid_accuracy` | Accuracy outside `[0, 100]` |
| `zero_time` | Elapsed time ≤ 0 seconds |
| `too_short` | Elapsed time < 5 seconds |
| `impossible_wpm` | WPM > 300 |
| `impossible_raw` | Raw WPM > 350 |
| `impossible_cps` | Characters per second > 30 |
| `impossible_burst` | Any single second exceeded 600 WPM |
| `flat_wpm_history` | All per-second WPMs within 1 WPM of each other over 4+ seconds |
| `perfect_consistency` | Consistency ≥ 99 at WPM ≥ 80 over 4+ seconds |
| `afk_detected` | More than 3 consecutive zero-raw-WPM seconds mid-test |

Do not weaken or remove these checks.

## Data Privacy and Storage

KeyZen has **no backend, no server, no analytics, and no remote data collection.**

### What is stored and where

| Data | Location | Persists across sessions |
|------|----------|--------------------------|
| Test mode, time/word/quote options | `localStorage` | Yes |
| Custom text input | `localStorage` | Yes |
| Code language and chapter | `localStorage` | Yes |
| Personal bests (`kz-pb-*`) | `localStorage` | Yes |
| Settings (theme, accent, font, sound) | React Context, re-read from `localStorage` on load | Yes |
| WPM history and per-test metrics | Memory only (React state) | No |

### What is never stored

- Keystrokes or typed text
- Test history beyond the current session
- Any data on a remote server
- Cookies

The only network requests the app makes are for static files: language word lists, sound packs, code snippets, and Google Fonts. All of these are public assets with no user identification.

### localStorage key reference

| Key | Type | Example |
|-----|------|---------|
| `tc-test-mode` | string enum | `"time"` |
| `tc-time-option` | number | `30` |
| `tc-word-option` | number | `50` |
| `tc-quote-length` | string enum | `"medium"` |
| `tc-punctuation` | boolean string | `"true"` |
| `tc-numbers` | boolean string | `"false"` |
| `tc-difficulty` | string enum | `"easy"` |
| `tc-custom-text` | raw string | `"..."` |
| `tc-code-language` | string | `"javascript"` |
| `kz-pb-{mode}-{detail}` | JSON object | `{"wpm":120,"accuracy":97,"date":"..."}` |

All `localStorage` reads are validated — invalid or out-of-range values are silently discarded and the default is used instead.

## PR Rules

- One concern per PR. Do not combine a bug fix with a refactor or a new feature.
- Run `pnpm typecheck`, `pnpm lint`, and `pnpm format` before pushing. PRs that fail these checks will not be reviewed.
- Do not edit generated files (`data/quotes.json`, `lib/code/data.json`, `public/sw.js`) directly. Generate them via the appropriate script.
- Do not add dependencies without discussion. Open an issue first if you think a new package is needed.
- Follow the no-`useEffect` rules from `.claude/skills/no-use-effect.md` when changing React code.
- Do not weaken or remove result validation or bot-detection logic.
- Sound pack additions must include a `config.json` for sprite-mapped packs, or a clean single `.ogg`/`.wav` for single-file packs. Verify the sound plays correctly in Firefox, Chrome, and Safari.
- Language additions must include a word list with at least 200 words, an entry in `_manifest.json`, and ideally a `_1k` variant.
- Screenshots or a short screen recording are appreciated for UI changes.
